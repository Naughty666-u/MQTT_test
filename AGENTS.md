# AGENTS.md

## Scope
- 本文件适用于当前仓库根目录及其所有子目录。
- 对本仓库进行代码修改、重构、补丁、注释补充时，默认遵循以下规则。

## Comment Policy
- 新增或修改代码时，默认补充中文注释。
- 注释重点解释“为什么这样写”和“这段逻辑的作用”，不要只重复代码字面意思。
- 简单赋值、明显的 getter/setter、机械性的初始化语句，不强制添加注释。
- 注释应简洁但足够清楚，优先帮助读代码的人快速理解数据流、状态变化和边界条件。

## Function Comments
- 新增函数默认补函数头注释。
- 函数头注释优先说明以下内容：
  - 该函数解决什么问题
  - 输入参数的业务含义
  - 返回值代表什么
  - 调用时机或调用频率
  - 是否会修改全局状态、硬件状态、队列、缓冲区
- 对外接口函数、任务函数、回调函数、状态机推进函数，函数头注释应比普通工具函数更详细。

## High-Value Logic
- 以下逻辑属于高价值逻辑，修改时必须补较为详细的注释：
  - 状态机
  - 串口收发流程
  - 环形队列或缓冲区读写
  - 中断回调
  - 定时任务/主循环任务
  - 事件检测与去抖
  - 阈值判断、滤波、识别流程
  - 网络上报、ACK、重试、超时控制
- 对这类逻辑，注释至少要说明：
  - 数据从哪里来，到哪里去
  - 状态如何切换
  - 为什么要限流、去抖、节流、清标志
  - 异常路径或超时路径如何处理

## Inline Comments
- 复杂分支、边界保护、长度检查、超时判断前，优先加一行短注释说明意图。
- 行内注释应解释设计目的，例如“避免重复触发”“防止缓冲区溢出”“保证字符串始终合法”。
- 避免出现“给变量赋值”“进入 if 判断”这类低价值注释。

## Embedded Style
- 注释要贴合嵌入式场景，优先说明：
  - 中断与主循环之间如何协作
  - 标志位由谁置位、由谁清除
  - 阻塞与非阻塞的区别
  - 硬件动作和软件状态是否一致
  - 时间单位、阈值单位、采样周期
- 涉及 UART、MQTT、继电器、电参采样、AI 识别时，注释要明确模块边界。

## When Refactoring
- 如果把主循环代码下沉到模块内，新的模块函数要补上“为什么要封装”的说明。
- 如果保留旧接口兼容，注释中要写清新旧语义差异。
- 如果某函数已基本废弃但仍保留，注释中应标明“遗留接口”或“兼容保留”。

## Preferred Style
- 优先使用块注释或短段落注释，不要求每行都注释。
- 注释风格以“先总后分”为主：
  - 先解释整段逻辑做什么
  - 再解释关键判断和关键状态
- 对复杂流程，优先写“步骤式注释”而不是零碎注释。

## Default Instruction For Codex
- 在本仓库中，如果用户没有特别说明，默认在代码修改时一并补充必要的中文注释。
- 如果用户要求“详细解释”或“补详细注释”，则应主动提高注释密度，覆盖函数头、关键分支和状态变化。
