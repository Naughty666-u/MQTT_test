# CH444 + 单串口四路 BL0942 分时采集改造说明

## 1. 目标与问题背景

### 1.1 目标
本次改造要解决以下工程目标：
- 在 MCU 仅使用 `UART3` 的前提下，同时采集并处理 4 路 BL0942 电参。
- 避免数据串路污染（例如 A 插座数据写入 B 插座状态）。
- 保持原有云端上报 / 网页控制链路不变，只替换底层采集调度方式。

### 1.2 原有单路方案的核心问题
原代码是“单通道固定采集”模式，主要表现为：
- `hal_entry.c` 中固定 `Data_Processing(..., 0)`，只更新 `socket[0]`。
- `g_uart3_rx_end` 由定时器置位，按时间切帧，容易在 9600bps 下提前截断 23 字节帧。
- 没有“当前采集通道”上下文，无法保证四路数据归属正确。

---

## 2. 硬件约束与映射关系

### 2.1 CH444 控制逻辑（依据你提供的电路）
- 选择输入：`IN1`、`IN0`
- 通道选择表：
  - `00` -> 第1路（S1A/S1B）
  - `01` -> 第2路（S2A/S2B）
  - `10` -> 第3路（S3A/S3B）
  - `11` -> 第4路（S4A/S4B）

### 2.2 代码中的引脚定义
在 `src/uart_hlw/uart_hlw.c` 中定义为：
```c
#define CH444_IN1_PIN BSP_IO_PORT_06_PIN_03
#define CH444_IN0_PIN BSP_IO_PORT_06_PIN_02
```

说明：
- `P603` 对应 `IN1`
- `P602` 对应 `IN0`

---

## 3. 总体改造思路

核心策略只有一句话：
**把“并行四路串口中断接收”改为“单串口分时轮询事务状态机”。**

单次采集事务流程：
1. 切换 CH444 到目标通道。
2. 清空本次帧缓存。
3. 发送 BL0942 读命令 `0x58 0xAA`。
4. UART 中断按字节收包，直到收满 23 字节。
5. 按当前通道调用 `Data_Processing(frame, channel)`。
6. 切换到下一通道。

这样能保证：
- 每帧采集都有明确 `active_channel`。
- 每次切通道前都清缓存，不带入上一路脏数据。
- 帧完整性由“字节计数”保证，而不是脆弱的“固定时间窗”。

### 4.2 新增 CH444 选择与采集状态机
文件：`src/uart_hlw/uart_hlw.c`

#### 4.2.1 关键常量与状态
```c
#define CH444_IN1_PIN BSP_IO_PORT_06_PIN_03
#define CH444_IN0_PIN BSP_IO_PORT_06_PIN_02

#define BL0942_FRAME_LEN         23U
#define BL0942_RX_TIMEOUT_MS     40U

typedef enum
{
    BL_STATE_IDLE = 0,
    BL_STATE_WAIT_FRAME,
} bl_poll_state_t;

static volatile uint8_t g_bl0942_rx_frame[BL0942_FRAME_LEN] = {0};
static volatile uint8_t g_bl0942_rx_count = 0;
static volatile bool g_bl0942_frame_ready = false;
static volatile bl_poll_state_t g_bl_poll_state = BL_STATE_IDLE;
static volatile uint8_t g_active_channel = 0;
static volatile uint32_t g_bl_deadline_tick = 0;
```

说明：
- `g_active_channel` 是防串路的关键上下文。
- `g_bl0942_rx_frame` 是“本事务独立缓冲区”。
- `BL0942_RX_TIMEOUT_MS` 用于坏帧快速跳过，避免卡死某一路。

#### 4.2.2 CH444 通道选择函数
```c
static void CH444_Select_Channel(uint8_t channel)
{
    /* channel: 0..3 -> IN1/IN0: 00/01/10/11 */
    bsp_io_level_t in0 = (channel & 0x01U) ? BSP_IO_LEVEL_HIGH : BSP_IO_LEVEL_LOW;
    bsp_io_level_t in1 = (channel & 0x02U) ? BSP_IO_LEVEL_HIGH : BSP_IO_LEVEL_LOW;

    g_ioport.p_api->pinWrite(g_ioport.p_ctrl, CH444_IN0_PIN, in0);
    g_ioport.p_api->pinWrite(g_ioport.p_ctrl, CH444_IN1_PIN, in1);
}
```

说明：
- `channel=0/1/2/3` 正好对应 `00/01/10/11`。
- 逻辑位直接映射到控制脚，代码清晰且不易写错。

#### 4.2.3 每次事务前清缓存
```c
static void BL0942_Rx_Reset(void)
{
    g_bl0942_rx_count = 0;
    g_bl0942_frame_ready = false;
    memset((void *)g_bl0942_rx_frame, 0, sizeof(g_bl0942_rx_frame));
}
```

说明：
- 这一步是防污染关键动作。
- 切到新路之前一定执行，避免上一帧残留。

#### 4.2.4 UART3 回调改为“按字节收满23字节”
```c
case UART_EVENT_RX_CHAR:
{
    if (g_bl_poll_state != BL_STATE_WAIT_FRAME)
    {
        return;
    }

    if (g_bl0942_rx_count < BL0942_FRAME_LEN)
    {
        g_bl0942_rx_frame[g_bl0942_rx_count++] = (uint8_t)p_args->data;
        if (g_bl0942_rx_count >= BL0942_FRAME_LEN)
        {
            g_bl0942_frame_ready = true;
        }
    }
    break;
}
```

说明：
- 只在 `WAIT_FRAME` 状态收字节，避免异步噪声干扰。
- 不再依赖旧的 `g_uart3_rx_end` 时间切帧。

#### 4.2.5 轮询状态机任务
```c
void BL0942_Poll_Task(void)
{
    uint32_t now = HAL_GetTick();

    switch (g_bl_poll_state)
    {
        case BL_STATE_IDLE:
        {
            /* 切通道 -> 清缓存 -> 发送命令 */
            CH444_Select_Channel(g_active_channel);
            BL0942_Rx_Reset();
            Send_com();

            g_bl_deadline_tick = now + BL0942_RX_TIMEOUT_MS;
            g_bl_poll_state = BL_STATE_WAIT_FRAME;
            break;
        }

        case BL_STATE_WAIT_FRAME:
        {
            if (g_bl0942_frame_ready)
            {
                uint8_t frame[BL0942_FRAME_LEN];
                memcpy(frame, (const void *)g_bl0942_rx_frame, BL0942_FRAME_LEN);

                /* 数据明确归属到当前路 */
                Data_Processing(frame, g_active_channel);

                g_active_channel = (uint8_t)((g_active_channel + 1U) & 0x03U);
                g_bl_poll_state = BL_STATE_IDLE;
            }
            else if ((int32_t)(now - g_bl_deadline_tick) >= 0)
            {
                /* 超时丢弃本路，转下一路，避免全局阻塞 */
                printf("[BL] timeout ch=%u rx=%u\r\n",
                       (unsigned)(g_active_channel + 1U),
                       (unsigned)g_bl0942_rx_count);
                g_active_channel = (uint8_t)((g_active_channel + 1U) & 0x03U);
                g_bl_poll_state = BL_STATE_IDLE;
            }
            break;
        }

        default:
            g_bl_poll_state = BL_STATE_IDLE;
            break;
    }
}
```

说明：
- 这是整个改造的核心。
- 每个循环只做“很短的一步”，适合裸机主循环。

#### 4.2.6 初始化阶段增加选择脚配置

## 5. 为什么这样做（设计理由）

1. 用状态机替代阻塞等待
- 原因：裸机主循环必须保持高频轮转，不能被串口等待或固定延时卡住。

2. 用“字节计数收满帧”替代“定时判帧结束”
- 原因：9600bps 下 23 字节耗时约 24ms，固定 20ms 计时容易截帧。

3. 每次切通道先清缓存
- 原因：CH444 是模拟开关复用，通道切换瞬间若不清缓存最容易串路污染。

4. 采集失败要超时跳过
- 原因：不能因为一路异常拖死其余三路。

---

## 6. 采样率评估

### 6.1 理论估算
- 9600bps，8N1：每字节约 10bit。
- 23 字节帧时间约：`23*10/9600 ≈ 24ms`。
- 四路轮询一圈：约 `4*24ms + 切换/处理开销`，实际通常 `100~130ms`。

### 6.2 结论
- 每一路“更新率”约 `8~10Hz`（接近原并行方案的 1/4）。
- 这属于分时复用的正常物理结果，不是代码错误。

## 8. 联调与验证步骤（建议按顺序）

1. 引脚与硬件验证
- 确认 `P603->IN1`、`P602->IN0`。
- 确认 CH444 `EN#` 为低有效且已使能。

2. 通道切换验证
- 在 `BL0942_Poll_Task()` 中打印 `g_active_channel+1`，确认 `1->2->3->4` 轮询。

3. 帧完整性验证
- 观察是否频繁打印 `[BL] timeout`。
- 若超时多，先把 `BL0942_RX_TIMEOUT_MS` 调大到 `50~60ms`。

4. 数据归属验证
- 分别给四路插座挂不同负载，观察 `sockets[0..3].power` 是否独立变化。

5. 云端链路验证
- 确认 `upload_strip_status()` 上报的四路 `power_w/device/on` 正常更新。

## 9. 当前已知注意事项

1. `timer1` 当前 FSP 配置仍可能是 20ms
- 如果目标是按键 10ms 扫描，需要在 FSP 里改周期并重新生成代码。

2. CH444 选择脚运行时已强制 `pinCfg`
- 推荐后续也在 FSP Pin 配置里同步登记，保持工程一致性。

3. 串口速率是系统上限因素
- 若要更高刷新率，可考虑提高 BL0942 UART 波特率（前提芯片和布线允许）。

---

## 10. 一句话总结

本次改造把系统从“单路硬编码采集”升级为“CH444 复用下的四路分时采集状态机”，核心收益是：
- 四路数据可稳定区分并入各自插座状态。
- 不再依赖脆弱的时间切帧，抗污染能力明显提升。
- 对现有网页/MQTT/AI 处理层保持兼容，改动集中且可维护。
