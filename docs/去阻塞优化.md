> 目标：先把会导致“卡死/长阻塞/命令丢失”的问题按风险排序，后续你可以按这个顺序逐步改代码。

## 0. 当前阻塞风险总览（先定优先级）

### P0（必须先做，直接影响“不卡死”）
1. MQTT 发送链路阻塞（ACK/status 发送等待 `>` 和 `OK`）
2. UART2 环形缓冲在阻塞期溢出，导致下行命令丢失
3. WiFi/MQTT 初始化失败后 `while(1)` 永久锁死

### P1（第二阶段，影响“及时响应”）
4. 继电器 100ms 脉冲为阻塞延时，拖慢主循环
5. SD 写入仍是同步阻塞（LEARN_COMMIT 虽已后台化，但写卡仍会拖时序）

### P2（第三阶段，影响“一致性与稳定体验”）
6. 总功率统计未按 `on` 状态过滤，网页展示冲突
7. 日志打印过于密集，串口带宽差时会放大阻塞

## 1. 第一步（P0-1）：把 MQTT 发送从阻塞改为非阻塞状态机

### 我们在解决什么问题
- 现在 `Send_Data_Raw()` 是同步等待流程，主循环会被单次发送卡住几百毫秒到数秒。
- 卡住期间不能及时处理新命令、不能及时 `get` 环形缓冲、BL0942 轮询也会变慢。

### 怎么定位问题（看哪些日志）
- 已有日志：
  - `[TX] RAW prompt timeout ...`
  - `[TX] RAW publish timeout ...`
  - `[PERF] cmd=... ack=... ms`
- 需要补的日志（建议）：
  - `[TXSM] state=WAIT_PROMPT elapsed=...`
  - `[TXSM] state=WAIT_OK elapsed=...`
  - `[MAIN] loop_max_cost=... ms`

### 改了哪些代码（文件/函数）
- `src/bsp_esp8266/bsp_wifi_esp8266.c`
  - `Send_Data_Raw()`（重构为状态机推进，不再整段阻塞）
  - 新增 `ESP8266_TxTask()`（每轮主循环推进一步）
- `src/bsp_esp8266/bsp_wifi_esp8266.h`
  - 新增发送任务接口
- `src/hal_entry.c`
  - 主循环加入 `ESP8266_TxTask()` 调度
- `src/cJSON_handle/cJSON_handle.c`
  - `send_ack()/upload_strip_status()` 改为“入队发送请求”，不直接阻塞等结果

### 关键修改代码（示例）

```c
/* 发送任务状态机：每轮主循环只推进一步 */
typedef enum {
    TX_IDLE = 0,
    TX_SEND_CMD,
    TX_WAIT_PROMPT,
    TX_SEND_PAYLOAD,
    TX_WAIT_OK,
} tx_state_t;

void ESP8266_TxTask(void)
{
    switch (g_tx.state)
    {
        case TX_IDLE: break;
        case TX_SEND_CMD:     /* 发 AT+MQTTPUBRAW */ break;
        case TX_WAIT_PROMPT:  /* 等 '>'，超时可恢复 */ break;
        case TX_SEND_PAYLOAD: /* 发 payload */ break;
        case TX_WAIT_OK:      /* 等 OK，失败重试 */ break;
        default: g_tx.state = TX_IDLE; break;
    }
}
```

### 为什么这样改
- 发送流程本质是异步串口交互，适合状态机；不适合同步 `while + delay`。
- 主循环只做“推进一步”，系统会持续响应其他任务。

### 最终效果如何
- 主循环不再被单次发送长时间占用。
- ACK 抖动显著减小；命令链路更稳定。

## 2. 第二步（P0-2）：修复 UART2 环形缓冲溢出导致的命令丢失

### 我们在解决什么问题
- 中断里 `g_rx_buf.put()` 失败（缓冲满）未处理，命令会静默丢失。
- 一旦上层阻塞，溢出概率急剧上升。

### 怎么定位问题（看哪些日志）
- 新增并观察：
  - `[RB] overflow uart2 count=...`
  - `[RB] high_watermark=...`
  - `[CMD] json_drop reason=overflow_or_timeout`
- 结合已有：
  - 命令下发但 MCU 无 ACK、无状态变化

### 改了哪些代码（文件/函数）
- `src/bsp_esp8266/bsp_wifi_esp8266.c`
  - `esp8266_uart2_callback()`：检查 `put` 返回值并累加溢出计数
- `src/circle_buf/circle_buf.c`
  - 增加可选“占用率统计/高水位统计”
  - 视内存情况把 `rx_buf` 从 1024 提升到 2048/4096
- `src/cJSON_handle/cJSON_handle.c`
  - `handle_uart_json_stream()` 增加异常帧重同步和丢帧日志

### 关键修改代码（示例）

```c
/* UART RX 中断里必须检查 put 返回值，避免静默丢包 */
if (g_rx_buf.put != NULL)
{
    if (g_rx_buf.put(&g_rx_buf, data) != 0)
    {
        g_uart2_rb_overflow_cnt++;
    }
}
```

### 为什么这样改
- 命令链路可靠性优先级最高；丢命令比慢更危险。
- 没有溢出计数就无法判断链路是否健康。

### 最终效果如何
- 从“静默丢命令”变成“可观测、可统计、可调参”。
- 在网络抖动或短时阻塞下，命令成功率明显提升。

## 3. 第三步（P0-3）：去掉初始化失败后的永久死循环

### 我们在解决什么问题
- `ESP8266_STA_JoinAP()`、`Connect_MQTT()` 等失败时 `while(1)`，设备永久卡死。

### 怎么定位问题（看哪些日志）
- 当前现象：设备卡在某个错误提示后不再输出主循环日志。
- 需要补的日志：
  - `[NET] init_fail step=... retry=... backoff=...`
  - `[NET] enter_degraded_mode`

### 改了哪些代码（文件/函数）
- `src/bsp_esp8266/bsp_wifi_esp8266.c`
  - `ESP8266_STA_JoinAP()`
  - `Connect_MQTT()`
  - `Subscribes_Topics()`
- 处理策略：
  - `while(1)` 改成“失败返回错误码 + 重试状态机 + 退避”

### 关键修改代码（示例）

```c
/* 错误改为可恢复返回，不再 while(1) 锁死 */
int Connect_MQTT(...)
{
    if (timeout_or_error)
    {
        return -1;
    }
    return 0;
}
```

### 为什么这样改
- 比赛现场/宿舍网络不稳定是常态，系统必须“可恢复”，不能“一错即死”。

### 最终效果如何
- 断网/连网失败时仍能继续本地控制和采样。
- 网络恢复后自动重连，不需要手动复位。

## 4. 第四步（P1-1）：继电器 100ms 脉冲改为“非阻塞脉冲任务”

### 我们在解决什么问题
- 当前 100ms 脉冲通过 `R_BSP_SoftwareDelay()` 实现，会阻塞主循环。

### 怎么定位问题（看哪些日志）
- `[PERF] cmd=ON/OFF ack=...ms` 出现固定台阶延迟。
- 新增：
  - `[RELAY] pulse_start socket=...`
  - `[RELAY] pulse_end socket=... cost=...ms`

### 改了哪些代码（文件/函数）
- `src/Relay/Relay.c`
  - `Relay_Set_ON()/Relay_Set_OFF()` 改为“置高并登记截止时刻”
  - 新增 `Relay_Task()` 到时拉低
- `src/Relay/Relay.h`
  - 暴露 `Relay_Task()`
- `src/hal_entry.c`
  - 主循环增加 `Relay_Task()`

### 关键修改代码（示例）

```c
/* 非阻塞脉冲：置高后登记截止时间，到时拉低 */
void Relay_Set_ON(uint8_t index)
{
    relay_ctx[index].active = true;
    relay_ctx[index].deadline = HAL_GetTick() + 100U;
    pinWrite(set_pin[index], HIGH);
}

void Relay_Task(void)
{
    if (relay_ctx[i].active && HAL_GetTick() >= relay_ctx[i].deadline)
    {
        pinWrite(set_pin[i], LOW);
        relay_ctx[i].active = false;
    }
}
```

### 为什么这样改
- 安全需求是“脉冲宽度 100ms”，不是“CPU 必须阻塞 100ms”。

### 最终效果如何
- 保持继电器安全驱动条件不变。
- 命令处理与采样不会被继电器动作打断。

## 5. 第五步（P1-2）：SD 写入任务再细化，避免长时间独占

### 我们在解决什么问题
- `AI_Commit_Task()` 已后台化，但真正 `f_open/f_write/f_close` 仍可能长阻塞。

### 怎么定位问题（看哪些日志）
- 已有：`[PERF] learn commit socket=... cost=... ms`
- 需要补：
  - `[SD] open_cost=... write_cost=... close_cost=...`
  - `[MAIN] loop_max_cost=... ms`

### 改了哪些代码（文件/函数）
- `src/Appliance identification algorithm/appliance_identification.c`
  - `AI_Commit_Task()` 增加“每轮只处理一个任务”与超时保护
- `src/sdcard_data_handle/sdcard_data_handle.c`
  - 可选：拆分成小步任务（打开/写入/关闭）

### 关键修改代码（示例）

```c
/* 每轮只处理一个 commit，削峰主循环耗时 */
void AI_Commit_Task(void)
{
    for (uint8_t i = 0; i < AI_SOCKET_NUM; i++)
    {
        if (!g_commit_task[i].valid) continue;
        (void)AI_Commit_Pending(i, ...);
        g_commit_task[i].valid = false;
        break;
    }
}
```

### 为什么这样改
- 减少主循环单次抖动峰值，比追求单次提交最快更重要。

### 最终效果如何
- 命令响应在 SD 慢卡情况下仍可接受。
- 写卡耗时可观测、可定位。

## 6. 第六步（P2-1）：修正总功率统计口径，避免网页冲突

### 我们在解决什么问题
- 单路 `power_w` 已按 `on` 过滤，但 `total_power` 仍累加所有路，可能出现展示矛盾。

### 怎么定位问题（看哪些日志）
- 对比 status 包：某路 `on=false`、`power_w=0`，但 `total_power_w` 偏大。

### 改了哪些代码（文件/函数）
- `src/uart_hlw/uart_hlw.c`
  - `Data_Processing()` 中总功率/总电流累加逻辑按 `g_strip.sockets[j].on` 过滤

### 关键修改代码（示例）

```c
float temp_p_sum = 0.0f;
float temp_c_sum = 0.0f;
for (int j = 0; j < 4; j++)
{
    if (!g_strip.sockets[j].on) continue;
    temp_p_sum += g_strip.sockets[j].power;
    if (g_strip.voltage > 1.0f)
    {
        temp_c_sum += (g_strip.sockets[j].power / g_strip.voltage);
    }
}
```

### 为什么这样改
- 统计口径必须和前端展示口径一致，避免用户误判“关了还在耗电”。

### 最终效果如何
- 网页数据显示一致，减少误报警与联调干扰。

### 关键修改代码（示例）

```c
#define LOG_THROTTLE_MS 3000U
if (HAL_GetTick() - last_log_tick >= LOG_THROTTLE_MS)

```

### 为什么这样改

### 最终效果如何

## 9. 每一步验收口径（建议固定）

- 命令 ACK 成功率：目标 >= 99%
- 命令到 ACK 耗时：常态 < 300ms，峰值可解释
- 状态上报间隔：心跳稳定，事件上报不抖动
- 环形缓冲溢出计数：持续为 0 或可控
- 网络异常恢复：无需重启可自动恢复
- 网页展示一致性：单路与总量不冲突

## 11. 本次已完成（P2-1）：修正总功率统计口径，避免网页冲突

### 我们这一步在解决什么问题
- 之前存在口径不一致：
  - 单路上报 `power_w` 在 `on=false` 时会归零；
  - 但 `total_power_w/total_current` 仍累计所有路瞬时功率。
- 导致网页可能出现“单路都关了但总功率不为 0”这类冲突显示。

### 怎么定位问题（看哪些日志）
- 重点看 status 包字段一致性：
  - `sockets[i].on=false` 且 `sockets[i].power_w=0`
  - 同时 `total_power_w` 仍异常偏大
- 可补充串口打印（可选）：
  - `[PWRCHK] on_count=... total_power=...`

### 改了哪些代码（文件/函数）
- 文件：`src/uart_hlw/uart_hlw.c`
- 函数：`Data_Processing()`
- 改动点：
  - 总功率/总电流累加时仅统计 `g_strip.sockets[j].on == true` 的通道
  - 总电流计算增加 `voltage > 1.0f` 保护，避免异常电压导致除零风险

### 关键修改代码（本次真实改动）

```c
/* Before */
for (int j = 0; j < 4; j++)
{
    temp_p_sum += g_strip.sockets[j].power;
    temp_c_sum += (g_strip.sockets[j].power / g_strip.voltage);
}
```

```c
/* After */
for (int j = 0; j < 4; j++)
{
    if (!g_strip.sockets[j].on)
    {
        continue;
    }

    temp_p_sum += g_strip.sockets[j].power;
    if (g_strip.voltage > 1.0f)
    {
        temp_c_sum += (g_strip.sockets[j].power / g_strip.voltage);
    }
}
```

### 为什么这样改
- 让总量口径与单路口径一致，前端展示才不会“自相矛盾”。
- 这是低风险改动，不影响识别状态机和命令链路，只修正统计层。

### 最终效果如何
- 网页上单路开关状态与总功率显示一致性提升。
- 可直接减少“误判漏电/关断无效”的联调噪声。
